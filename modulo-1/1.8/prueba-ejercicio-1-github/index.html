Introduccin

Hasta ahora hemos hecho una primera aproximacin a Git, nada menor. Sabemos crear un repositorio y subir y bajar cambios a travs de:

git add -A
git commit -m "Message commit"
git push

Tambin sabemos cmo publicar nuestro trabajo a travs del sistema de hosting de GitHub: GitHub Pages.

Hoy vamos a ver cmo mejorar nuestros repos y trabajar en grupo sobre el mismo proyecto y sus archivos.

Mejorando nuestros repositorios

A continuacin vamos a aprender 3 ficheros que usamos en los repositorios para gestionarlos mejor y aadir informacin til para las personas que vean nuestros repos.

README.md

El archivo README.md es un documento escrito en markdown que se presenta en la pgina principal del repositorio y tiene como objeto aportar una primera documentacin o presentacin del proyecto.

Markdown es un lenguaje de marcado como HTML pero ms simple. Al ser ms simple y permitirnos menos opciones, nos resulta ms fcil de aprender y adems todos los proyectos que hay en GitHub tiene un formato comn.

Ejemplos de readme.md:

  • React

  • Editor de cdigo VS Code

  • Gulp (Automatizador de tareas)

Os recomendamos que cuando tengis tiempo le pongis un buen README.md a todos vuestros repositorios. Las empresas que estn buscando Adalabers lo agradecern.

.gitignore

Git tiene un sistema para poder ignorar archivos de un repositorio. Y por qu querramos hacer esto? Porque hay archivos que necesitemos en nuestra carpeta de trabajo local pero que no queremos subir al repositorio ni controlar sus cambios.

En proyectos pequeos como los que tenemos ahora vamos a querer siempre subir al repositorio remoto todos nuestros archivos. Pero pronto pasar que tendremos una carpeta con los archivos de diseo de ciertas partes del proyecto o archivos comprimidos que usamos para enviar a nuestro cliente los avances.

NOTA: Es muy comn que el propio sistema operativo cree en cada carpeta una serie de archivos o carpetas ocultas que le ayudan a realizar tareas como la bsqueda de archivos. Estos archivos del sistema operativo no queremos subirlos al repositorio remoto.

Estos archivos, que no tiene sentido tener "controlados" pero que en nuestra repo local queremos mantener, se listan en este archivo especial, .gitignore para decirle al algoritmo de Git que no los tenga en cuenta.

En gitignore.io podemos encontrar una serie de configuraciones ya hechas que nos ayudan a ignorar tipos de archivos comunes segn el sistema operativo o el lenguaje en el que trabajemos.

Licencia

Uno de los puntos claves en un entorno social donde poner al alcance de todos tus proyectos, es indicar cmo y en qu trminos se deben usar. Para esto estn las licencias, que son archivos legales que especifican qu se puede y qu no se puede hacer con los archivos asociados.

GitHub nos ofrece un enlace donde nos intenta orientar sobre qu licencia elegir en cada caso: choosealicense.com.

Compartir cdigo con mi equipo

Otras de las bondades de Git es que hace que trabajar en grupo sea seguro y ms fcil. Nos evita por ejemplo: tener que compartir archivos a travs del email, perder cdigo cuando una compaera lo sobreescribe por error...

Las primeras semanas de utilizar Git nos equivocamos de vez en cuando y creemos que hemos perdido un montn de cdigo que nos ha costado la vida crear. Y lo que es peor, a veces nos pasa eso con el cdigo de nuestras compaeras. Pues gracias a Git todo lo que est dentro de un commit no se pierde nunca. Git nos permite viajar al pasado, recuperar el cdigo y volver al presente.

El primer paso es permitir a nuestras compaeras modificar nuestro repo. Desde el GitHub de nuestro repositorio accedemos a Settings y desde all a Manage access donde podremos aadir a nuestras colaboradoras favoritas, o a las que nos toquen ;)

Modificar archivos en local y subir a remoto los cambios

Ya lo hemos ido viendo estos das:

  1. Se modifican archivos y se guardan los cambios en el editor.

  2. Se aaden para su control con git add -A (aade todos los archivos modificados) o con git add nombre-de-archivo (aade solo el archivo especificado).

  3. Creamos el commit con git commit -m "Short description of the commit"

Hasta aqu todo normal. Ahora llega el momento de subir el commit (o los commits, si hemos hecho varios) con nuestros cambios al repositorio remoto con git push origin master, pero pueden pasar varias cosas:

  1. Que se suba bien, sin problemas ni conflictos. Yay!

  2. Que no podamos subir nuestros cambios porque no estemos trabajando con la ltima versin. Lo sabremos porque la terminal nos muestra un error. En este caso tendremos que hacer git pull para actualizar nuestro repositorio local con la ltima versin que se encuentra en remoto, es decir, nos traemos los ltimos cambios hechos por otras personas a nuestro ordenador.

NOTA: Antes de comenzar a trabajar, de vez en cuando y antes de hacer un commit es una buena prctica hacer git pull y actualizar nuestro repositorio local con los cambios que otras personas han subido al repositorio remoto. Aun as, ocurrir que tras hacer nuestros cambios y commitearlos, al intentar hacer git push el terminal nos indique que tenemos que hacer git pull primero, esto ocurre por que alguna compaera ha subido cambios mientras nosotras trabajbamos.

Qu pasa cuando hacemos un git pull?

Pasan varias cosas:

  1. Git comprueba si en el repositorio remoto hay una versin posterior (ms nueva) a la que se encuentra en nuestro repositorio local.

  2. Si encuentra cambios posteriores, se los baja e intenta mezclarlos con los de nuestros commits.

Y aqu tenemos dos escenarios diferentes:

  1. Los cambios que se han bajado de remoto (realizados por mis compaeras) y los mos se pueden mezclar (o mergear) automaticamente, Git crea un commit automtico con esta mezcla (o merge) y nos lo muestra usando el editor por defecto que tenemos configurado en nuestra terminal (NANO, vim...). A esto se le llama un soft merge.

  2. Otra posibilidad es que Git no pueda mezclar los cambios automticamente. Entonces nos avisa de que hay conflictos que tendremos que resolver nosotras manualmente. Nos mostrar una lista de archivos donde se encuentran los conflictos. Es importante leer toda la info que nos muestra la terminal. A esto se le llama un hard merge.

NOTA: En el primer caso podremos cambiar el mensaje del commit automtico o poner uno nuevo. Guardamos aceptando el nombre que nos propone, salimos, y hacemos un push (se subir el commit con nuestros cambios y el commit con el merge o mezcla).

Qu pinta tiene un conflicto?

Un conflicto ocurre cuando git se encuentra con dos versiones del mismo bloque de cdigo. Entonces, marca en el documento que hay un conflicto y muestra las dos opciones para que nosotros elijamos qu hacer:

<<<<<<<
1 versin del cdigo en conflicto
=======
2 versin del cdigo en conflicto
>>>>>>>

<<<<<<<: Indica el inicio de la zona de conflicto, en la lnea siguiente muestra el primer bloque en conflicto.

=======: Separa las dos versiones, seguidamente muestra el bloque alternativo que est dando conflicto.

>>>>>>>: Indica el final e la zona de conflicto

Aqu puede pasar que queramos la primera opcin, la segunda, las dos, o una mezcla de las dos.

La manera de afrontar este conflicto es elegir lo que queremos que ponga en ese bloque, quitar los separadores que aade Git, guardar el archivo y hacer add, commit y push con normalidad.

Los conflictos ms pequeos los resolveremos sobre la marcha, en los ms complicados tendremos que hablar con la compaera que haya hecho los cambios para decidir qu hacer. No queremos machacar el cdigo de una compaera sin su permiso.

Ramas

Cuando subimos los commits habris visto que escribimos $ git push origin master, lo que estamos diciendo es que suba muestra rama master al repositorio remoto.

Madre de CROM, qu es una rama???

Git nos permite crear versiones paralelas de nuestro proyecto para poder desarrollar o probar varias funcionalidades a la vez sin miedo a perder lo hecho hasta ahora:

Cuando iniciamos un repositorio git se crea una primera rama, y se llama master por convencin. En la ltima sesin trabajamos en esa rama.

Vamos a ver el trabajo en ramas a travs de un ejemplo, como un mini proyecto de grupo, porque al fin y al cabo, Git va de trabajar en grupo.

EJERCICIO 1

  1. Vamos crear un repositorio por pareja, donde ambas debis tener acceso al repositorio (la que lo crea debe dar acceso al usuario de GitHub de la otra).

  2. Crearemos una primera versin de nuestra web (solo en HTML) que tendr:

    1. Un <header> con un <h1> con el nombre del grupo

    2. Un <main> con dos secciones:

      2. <section class="contenido"></section>

    3. Un <footer> con un <p> con el texto: "Maquetado en grupo en Adalab"

  3. Lo subiremos a GitHub

Nos tiene que quedar algo as:

Creando ramas

Para crear ramas escribimos y movernos a ella tenemos dos comandos:

git branch nombre-de-la-rama
git checkout nombre-de-la-rama

Tenemos un atajo para crear la rama y cambiarnos a ella directamente:

git checkout -b nombre-de-la-rama

En cualquier caso, si queremos movernos de una rama a otra usaremos git checkout nombre-de-la-rama, de esta manera podemos movernos a nuestra nueva rama o volver a master en cualquier momento.

NOTA: Para poder movernos entre ramas debemos tener todos los archivos modificados, al menos, aadidos a un futuro commit. Si modifico un archivo y me intento cambiar de rama no me dejar.

Aadir archivos y crear un commit funciona igual pero cuando queramos hacer un push usaremos:

git push origin nombre-de-la-rama

La primera vez usaremos el git push con -u:

git push -u origin nombre-de-la-rama

EJERCICIO 2

  1. Una de la pareja crear una rama footer, nos movemos a ella y modificamos un poco nuestro proyecto. Aadiremos a nuestro footer el enlace a la web de Adalab, quedando as:

    <footer>
        <p>Maquetado en grupo en <a href="http://adalab.es">Adalab</a></p>
    </footer>

  2. Como siempre, aadimos, commiteamos y hacemos push, esta vez usando git push -u origin footer.

  3. Si ahora cambiamos a la rama master veremos que permanece como la dejamos y que el cambio del enlace solo est hecho en nuestra rama footer.

Fusionar ramas

Una vez que hemos terminado el trabajo en nuestra nueva rama y lo hemos subido al servidor remoto queremos aplicar estos cambios en nuestra rama principal, master.

Para ello nos vamos a la rama destino (en este caso master) con git checkout master, y escribiremos:

git merge nombre-de-la-rama

Esto nos mezclar nuestra versin local de la rama nombre-de-la-rama con la rama donde estemos, en este caso, master. Si todo va bien nos mezclar las ramas, crear un commit automtico y si hacemos un git status nos dir que solo queda hacer un git push origin master y ya.

NOTA: Es importante haber hecho un git pull en la rama que vamos a fusionar, en este caso nombre-de-la-rama antes de empezar el proceso de fusin para asegurarnos de que tenemos la ltima versin en ambas ramas.

EJERCICIO 3

Vamos a fusionar nuestra rama footer con master para que nuestra web tenga el enlace que hemos aadido anteriormente. Para ello:

  1. Nos movemos a la rama footer.

  2. Comprobamos que est correcto y tenemos la ltima versin.

  3. Nos movemos a la rama master (s, es super buena idea asegurarnos de que tambin tenemos la ltima versin).

  4. Hacernos un merge de la rama footer.

  5. Resolvemos los conflictos si los hay.

  6. Comprobamos que los cambios est hechos.

  7. Y subimos al repositorio remoto.

<footer>
    <p>Maquetado en grupo en <a href="http://adalab.es">Adalab</a></p>
</footer>
